Electron

Building Electron from the source shows the build arguments do not have Control Flow Integrity enabeld. See this line.
Control Flow Integrity (CFI) is an exploit mitigation that aims to prevent altering of the flow of a program or code reuse attacks. Chromium by default has Control Flow Integrity enabled, Linux and Android using Clang's forward-edge CFI, Windows using Control Flow Guard.

Suggestion: Use your software, such as Discord, in Google Chrome or any well-known Chromium browser. They're built with security in mind, and even if your Electron program grossly manages memory (looking at you Discord with old Electron builds), the browser can attempt to do a better job.

Unprivileged User Namespaces

There are both pros and cons to SUID sandboxing and User Namespaces, but it realistically depends on your threat model. SUID essentially drops privileges while user namespaces isolate identifiers and attributes such as user and group IDs, root directory, kernel capabilities, etc. This is great and the only way to isolate containers by using user namespaces. However containers suffer a major trade off forcing you to isolate them (unprivileged user namespaces) at the cost of a huge attack surface layer in the kernel. If you don't run Docker with user namespaces, it runs it as root. If you don't have mandatory access control (e.g. SELinux) with say Podman or Docker, you have kernel and root privileges as long as you have the daemon access.
There is a Docker experiment to being able to run it without root, but it's far from complete.
Another thing is user namespaces are getting popular. Malware will cater towards people with user namespaces enabled instead of SUID. Containers are popular too.

Suggestion: Use SUID sandboxing if you do not deal with containers at all. If you need containers, use them with SELinux.
